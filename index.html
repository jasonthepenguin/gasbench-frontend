Model Leaderboard

Each model was tested on multiple customer service refund scenarios. Note: All models tested are non-reasoning models. "API calls" shows the total number of API calls needed to complete all tasks. "Non-failing %" represents the percentage of function calls that executed successfully without errors.

Why fewer turns with higher reliability is a strong signal

  • Fewer turns = better planning: Completing days in fewer API calls shows the model can reason upfront, choose the right actions, and avoid unnecessary back-and-forth.
  • Policy and constraints adherence: High non-failing rate means it checks refund policy, capacity, and cash before acting—reducing wasted attempts and retries.
  • Effective action ordering: Prioritizing the right calls (e.g., validate receipt before refund, check status before ordering) prevents avoidable failures.
  • No loops or churn: Low total actions with few (or zero) failed actions indicates it doesn't get stuck correcting itself or repeating mistakes.
  • Operational impact: This combo yields faster runs, lower cost, cleaner logs, and more consistent profits—exactly what your leaderboard measures.

Key Differences

  • Efficiency: Sonnet completes the task in 4 API calls vs. 6 for GPT-4o
  • Strategy: Sonnet checks inventory first, then verifies receipt; GPT-4o attempts refund immediately and fails
  • Error handling: Sonnet avoids errors entirely; GPT-4o has 2 failed attempts
  • Customer experience: Both complete the refund, but Sonnet's more efficient approach means faster resolution
